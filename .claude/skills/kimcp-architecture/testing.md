# Testing

Testing is part of the architecture, not a side-task. The coverage-matrix rule in `tool-catalog.md` and the backend-compatibility matrix in `backends.md` are enforced through the suite described here.

## Layers

1. **Unit tests** — pure Python logic (schema generation, rule engines, dispatcher selection).
2. **Property-based tests** — rule engines and S-expression parsers with hypothesis-driven inputs.
3. **Golden-file tests** — S-expression parser round-trip fidelity against fixture `.kicad_sch` / `.kicad_pcb` / `.kicad_sym` / `.kicad_mod` files.
4. **Integration tests** — real `kicad-cli`, real SWIG `pcbnew`, real IPC API against a running KiCAD instance.
5. **Matrix tests** — (KiCAD version × OS × backend) grid.
6. **End-to-end tests** — full MCP sessions with a scripted client, exercising transport + tool + backend + resources.
7. **Benchmark tests** — latency budgets from `performance.md`; regressions fail CI.
8. **Fuzz tests** — S-expression parser, schema validation, input-path handling.

## Coverage expectations

- No PR merged if coverage (line + branch) drops on touched modules.
- The KiCAD feature → tool coverage matrix (`tests/coverage_matrix.md` in the impl repo) is a checked-in file; CI diffs it against a reference list of menu items / `kicad-cli` subcommands and fails on regressions.
- At least one real-backend integration path exercised per tool — mocking all four backends simultaneously is forbidden.

## Fixtures

- `tests/fixtures/projects/` — small, medium, large KiCAD projects with known properties (component counts, nets, DRC violations, etc.).
- `tests/fixtures/datasheets/` — a handful of redistributable datasheets for datasheet-search tests.
- `tests/fixtures/footprints/` — deliberately-broken footprints to test validators.
- `tests/fixtures/snapshots/` — expected outputs for exports (Gerber, PDF, STEP); compared by content hash not byte-identity (some outputs have timestamps).

## Golden-file strategy

For S-expression parser: `read → modify → write → re-read → re-modify → write`. Result must equal first-write byte-for-byte for untouched sections. Touched sections compared structurally (order-insensitive where the format allows).

## Property tests (hypothesis)

- S-expression parser: round-trip identity for random well-formed inputs.
- Schema generation: generated JSON Schema validates the Pydantic model's own examples.
- Rule engine: applying rules is idempotent (running twice produces the same suggestions).
- Dispatcher: for a fixed backend-availability set, selection is deterministic.

## Integration-test matrix

Per ADR-0014, the default matrix targets KiCAD 9+ only. Community-contributed back-port rows may appear behind a `compat` flag but are not part of the default support commitment.

```
KiCAD versions:  9.x (IPC + CLI + sexpr + SWIG), 10.x (IPC + CLI + sexpr + SWIG), nightly
OS:              macOS 13+, Ubuntu 22.04+, Windows 10+
Backends:        IPC, CLI, sexpr, SWIG
Compat (opt-in): 8.x (CLI + sexpr + SWIG, no IPC), 7.x (CLI + sexpr + SWIG, no IPC) — community-maintained
```

Not every combination runs on every PR. On-push: one representative tuple per OS against KiCAD 10.x. Nightly: full 9.x + 10.x matrix. Pre-release: full matrix + benchmarks. Compat rows run only when `KIMCP_COMPAT_MATRIX=1` is set.

## Destructive-op tests

Every `destructive` tool has:
1. A happy-path test asserting snapshot is created and referenced in `meta.snapshot_ref`.
2. A refusal test confirming it blocks without a confirmation token.
3. A rollback test using `restore_snapshot` to verify the pre-change state returns bit-identical.

## Error-path tests

- Tools assert sensible JSON-RPC errors for missing backends, permission failures, malformed inputs.
- Errors carry `rule_id` and `details` where applicable.
- No bare 500s — every exception path maps to a typed error.

## Fuzz corpus

- `hypothesis` + stored corpora for:
  - S-expression parser input
  - Pydantic validation on every tool
  - Path-sanitization logic (see `security.md`)
- Nightly fuzz runs with corpus growth; crashes file GitHub issues automatically.

## Simulation-dependent tests

- ngspice availability probed per platform; tests that require it are skipped with a clear reason on platforms where it's absent.
- IBIS model tests use redistributable vendor IBIS files committed to the repo.

## Benchmarks

Per `performance.md` budgets. Runner:

- Runs benchmarks isolated on a consistent machine profile (documented in `bench/README.md`).
- Stores results as JSON per commit.
- CI compares current run to a rolling baseline; > 10% regression on any named benchmark fails the run unless the PR includes `perf-regression-approved` label with justification.

## Anti-patterns

- Tests that mock the dispatcher and all backends — they verify nothing real.
- Tests whose fixtures are generated by the code under test — they can't fail usefully.
- Tests depending on user's installed KiCAD version silently — always gate explicitly.
- Tests that write outside `tests/tmp/` — they leak state.
